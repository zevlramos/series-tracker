# The wizard has a first-class direct-interaction override — a maintainer-chosen detour that always re-enters the publish gate

The Curation wizard is the default control surface, but it is **UI-shaped**: it expresses placement, inclusion, reasons, and ranks one Entry (or one previewable permutation) at a time. Some maintainer asks don't fit that shape — a **global/relational ordering rule** ("interleave all games + main-cast films in lore order, then everything else in lore order"), a **bulk semantic edit across many Entries**, or a **constraint spanning Entries a lens can't cover**. Faced with one of these, the maintainer either grinds through dozens of per-Entry actions or feels trapped in the UI. This decision makes a **direct-interaction override** a first-class, documented step: when the wizard can't express an ask, Claude drops to a conversational/deterministic channel, does the work against the **Draft**, and hands control back to the wizard's `parseSeries` Publish gate — so the override is *more* control, never *less* validation. Surfaced throughout [#41](https://github.com/zevlramos/series-tracker/issues/41) (the Resident Evil re-curation), where the maintainer repeatedly needed precision the wizard couldn't give and explicitly asked for "a direct-interaction fail-safe to override and get more specific levels of control."

## The decision

The override is a **detour, not a fork**: the maintainer leaves the wizard, the deterministic work happens against `.drafts/<slug>.json`, and the maintainer **reloads the wizard to rejoin** and Publishes normally. It is fixed on five points:

- **TRIGGER — an ask the wizard can't express as one action; Claude offers, the maintainer chooses.** The qualifying shapes are: a *global/relational ordering rule* (a whole-set order derived from a stated rule, not a per-Entry nudge), a *bulk edit across many Entries* (the same transform applied N-wide), or a *constraint spanning Entries a lens or chip can't reach* (e.g. a spinoff absent from every researched lens). Claude **recognizes the friction and offers** the override; the maintainer always chooses to drop. The UI stays the default — Claude **never silently bypasses it** (the [ADR-0013](0013-fandom-order-refusable-suggestion-lenses.md)/[ADR-0015](0015-refusable-reason-suggestions-proposed-producer.md) "maintainer-initiated, never silent" shape, applied to the control surface itself).

- **WHEN — the moment it's noticed, via pause → state → edit → reload → continue.** The wizard **autosaves the full working set *down* to the Draft** on every edit, and the **browser's in-memory copy is the source of truth while the tab is open** (`curate-server.mjs` only re-reads the Draft on a page load's `GET /data`). So the protocol is forced by the architecture: (1) the maintainer **pauses** — stops touching the wizard; (2) **states the ask** in chat; (3) Claude does the work **against the Draft file** (directly, or via a subagent — see SUBAGENT); (4) the maintainer **reloads the wizard**, which re-`GET`s `/data` and pulls the edited Draft into the browser — *this reload is the rejoin point and is mandatory*; (5) the maintainer **continues and Publishes** normally. This is the same "reload `localhost:8123`" step the maintainer performed during #41.

- **GATE — the override edits the Draft, never `data.json`; re-entry is reload + Publish through the same `parseSeries`.** Because the override only ever writes the **Draft** (ungated scratch, the `/stage` write model), it is *structurally* incapable of skipping validation — the published file is still written only by the wizard's `POST /publish`, which projects through `draftToSeriesData` and the fail-closed `parseSeries` gate ([ADR-0008](0008-draft-is-structured-durable-review-enriched.md), [ADR-0012](0012-unified-curation-pipeline.md)). "Reload + Publish" *is* the gate re-entry; there is nothing extra to bolt on. A **headless fallback** is documented for when the wizard server isn't running at all: call `draftToSeriesData(draft)` then `parseSeries(...)` directly and write only on `ok` — the *same gate function*, never a hand-written `data.json`. Both paths end at `parseSeries`; direct-to-`data.json` is forbidden under either.

- **HAZARD — the autosave race; "pause, then reload" is the safety rule.** The one way the override corrupts state is if the maintainer keeps editing in the browser while Claude's Draft write is in flight: a pending debounced `/stage` (or any post-edit autosave) carries the browser's *stale* in-memory copy back down over Claude's file edit. The rule that defuses it is the protocol's own spine — **stop touching the wizard during the override, and reload to rejoin** — so the browser's authoritative copy is *replaced by* the edited Draft rather than *racing* it. The override is never applied silently underneath a live, still-being-edited tab.

- **SUBAGENT — heavy deterministic work goes to a focused subagent; Claude audits before applying.** When the override is a computation (e.g. deriving a full Recommended order from a stated rule across 76 Entries), Claude **dispatches a focused subagent** to produce it and then **audits the result for quality before it touches the Draft** — the maintainer is not asked to trust an unreviewed machine ordering. This keeps the deterministic muscle off the main thread while preserving the *authored* contract ([ADR-0011](0011-curated-chronological-rank-lore-date-enrichment.md)): a human (Claude on the maintainer's behalf, then the maintainer at the reload-rejoin and Publish) vouches for what lands.

## Why a general override instead of more wizard features

The sibling wizard-gap issues each built a *specific* affordance — refusable reason suggestions ([#64](https://github.com/zevlramos/series-tracker/issues/64)/[ADR-0015](0015-refusable-reason-suggestions-proposed-producer.md)), a bulk "apply lens as baseline" ([#65](https://github.com/zevlramos/series-tracker/issues/65)/[ADR-0013 amendment](0013-fandom-order-refusable-suggestion-lenses.md)), deleting a dead button ([#63](https://github.com/zevlramos/series-tracker/issues/63)). Those close the *known* gaps. The override is the **general escape hatch for the unknowable ones** — the maintainer ask no one anticipated and no single button expresses. It is the meta-principle behind all three: a maintainer should be able to drop to direct interaction for *any* ask rather than wait on a UI feature, while the publish gate stays authoritative. Encoding it as a skill principle (not ad-hoc per session) keeps the override consistent and discoverable.

## Considered options

- **Build a wizard feature for every ask** — rejected. The space of relational/global rules is open-ended; you cannot pre-build a button for every ordering constraint a maintainer might state. The override is the general fallback that makes the wizard's expressiveness a floor, not a ceiling.
- **Let Claude silently edit the Draft beneath a live wizard tab (no pause/reload protocol)** — rejected. The autosave-down architecture guarantees the browser's stale in-memory copy clobbers the edit on the next `/stage`. Pause-then-reload is not ceremony; it is the only race-free rejoin.
- **Write `data.json` directly in the override** — rejected. It skips `parseSeries`, the single fail-closed gate ([ADR-0012](0012-unified-curation-pipeline.md)). The override edits the *Draft* precisely so validation can never be bypassed; the published write stays the wizard's (or the headless gate call's) job.
- **Make the override an always-available default surface** — rejected. The wizard is the default for a reason (refusable suggestions, gate-backed structure); a standing "just edit the file" escape erodes that. The override is a *deliberate, offered detour* for asks the UI can't express, not a parallel everyday path.
- **An interactive grill/spec session per override** — rejected as overkill for the in-the-moment case. The override is meant to be reached *during* curation, in seconds; the protocol (pause → state → edit/audit → reload → Publish) is the lightweight channel. Heavier design work still belongs in the idea→ship flow, not mid-wizard.
