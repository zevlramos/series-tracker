# Fandom-order help is a refusable suggestion (pre-baked lenses), never a seed

The Curation wizard's Order phase helps the maintainer author the **Recommended order** by offering researched orderings as **refusable suggestions**, not by writing the order for them. Nothing is written to `recommendedOrder` until the maintainer explicitly accepts (per Entry) or drags — so the field stays *authored, never derived* ([ADR-0011](0011-curated-chronological-rank-lore-date-enrichment.md)), and a suggestion can never clobber preserved curation on an update.

The help is delivered as **pre-baked lenses**: a research pass writes a few candidate orderings ("lenses") into the Draft up front — **Release order** (deterministic, computed from `releaseDate`, always present as the honest floor), a **Fan-consensus** order (present only when real consensus exists), and 0..n **Alternative framings** (present only on a real, sourced split). The wizard switches between them instantly. Lenses suggest a **permutation of the current included set** only — positions, never add/remove (relegating an Entry is a *removal*, which belongs to the Include phase). They render as ghost-chip deltas against a baseline that differs only by entry point — **Release order for a create, the maintainer's preserved order for an update** — keeping the wizard identical for both ([ADR-0012](0012-unified-curation-pipeline.md)). All lens data is `_`-prefixed Draft-scratch, stripped at publish; nothing new reaches `data.json`. Three honesty states are handled explicitly — *contested* (a split to adjudicate), *uncontested* (one obvious order, no fabricated debate), *thin/no-signal* (degrade to the baseline and say so) — so the tool never invents authority.

## Considered options

- **A seed that writes `recommendedOrder` before the wizard** — rejected. On an update it overwrites the maintainer's preserved order (the [ADR-0009](0009-stable-content-derived-ids-semantic-matching.md) curation-preservation contract); it also treats the recommended order as derived rather than authored. The suggestion model makes clobbering structurally impossible.
- **A "smart" server that calls an LLM on each re-steer click** — rejected. It puts a live model dependency behind a curation UI and complicates `curate-server.mjs`, which is deliberately a dumb file-server. Pre-baking a few lenses up front gets refusability and re-steering with zero live calls.
- **An orchestrator round-trip for free-text re-steering** — deferred, not rejected. Only worth it if open-ended "re-research with this constraint" steering is ever wanted; the pre-baked lenses cover the known framings without it.
- **Auto-applying the suggestion on create only (a hybrid)** — rejected. It saves one click but reintroduces an `if (create)` branch into a wizard whose whole point is not knowing create from update. Pure suggestion keeps one uniform surface.
- **Mainline-first as the baseline ordering** — rejected/retired. The old deterministic seed grouped all mainline Entries ahead of all spinoffs regardless of date, which is the very arrangement that stranded mid-timeline spinoffs (e.g. Resident Evil Outbreak after a far-later mainline Entry). Release order is a more honest, data-backed baseline and removes that stranding at the root.
