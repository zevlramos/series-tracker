# Versions of one work are a durable group; deliberate exclusions are remembered, not deleted

Two facts that curation produces must now **survive the publish → re-import cycle**, so each becomes a minimal additive field on the Entry. This amends [ADR-0007](0007-remakes-are-curation-not-schema.md), which kept both out of the schema on the explicit grounds that "no Series requires it" and that the field is "additive, free to defer." Resident Evil now requires it (the 1996 / 2002 / 2015 versions of RE1, plus originals deliberately not shown), so we cash in the deferred option. Everything that *could* have carried these facts before — `versionNote`, the `_pairings` scratch, the Include-phase `_drop` decision — is stripped by the 14-field publish whitelist, so there is no scratch-only fix: something has to become durable.

**`versionGroup` — a shared, content-derived slug linking the versions of one work.** A **Version group** (CONTEXT.md) is the set of Entries that are alternative versions of one underlying work. It is *not* a standalone record and *not* a shared id — each member keeps its own unique `id` and additionally carries a `versionGroup` string whose value is shared across the members (`re1-original` / `re1-remake` / `re1-remaster` all carry `versionGroup: "re1"`). The group is emergent: "the entries where `versionGroup === 're1'`," materialized by a `group-by` when the version card or a redundancy check needs it. The slug is minted once and stable, same philosophy as Entry ids ([ADR-0009](0009-stable-content-derived-ids-semantic-matching.md)); we do not infer grouping by parsing id prefixes. `versionGroup` is a **curation field**, preserved across `update-series` runs so re-research can never dissolve or reshuffle an established group — but because the merge restores curation fields only onto *existing* entries, research may still propose a `versionGroup` on a genuinely-new entry (a freshly-released remaster of an already-tracked work), seeding the card. This is the durable replacement for the old title-matching pairing (`stripYear` + base-title equality + a publish-stripped `versionNote`), which produced zero pairings on update; pairing is now a trivial group-by and scales from two members to N with no new logic.

**`excluded` — a deliberate "known but not shown" decision, retained rather than deleted.** Marking an Entry **Excluded** (CONTEXT.md) keeps it in `data.json` with `excluded: true` instead of dropping it. This is a **general per-Entry property**, not version-scoped: any deliberately-dropped Entry — a non-canonical version, a not-worth-tracking tie-in, or a research false positive — is retained-excluded, with the reason varying but the durable fact uniform. `excluded` is a curation field, never flipped by re-research. The reader never sees an excluded Entry: `parseSeries` keeps it (so `update-series` can load and preserve the memory), and the **Shell** filters it out at its single entry-consumption chokepoint, removing it from the Timeline, progress counts, and every Lens. This makes the re-surfacing bug fall out of [ADR-0009](0009-stable-content-derived-ids-semantic-matching.md) for free: because the excluded Entry stays in `data.json`, the next `update-series` re-research aligns its re-discovered candidate onto the existing id and it lands in `changed`/`unchanged`, **never `new`** — the exclusion is remembered, not re-litigated every run.

**What stays out of the schema.** The fine-grained lineage role (original / remake / remaster / port) is **not** a durable field — it is research output that labels the card and dies at publish like `_pairings`/`_orderResearch`. Group membership (durable) is all redundancy detection needs; ordering within a group derives from `releaseDate`; so the durable surface stays the minimum that closes the gaps: `{versionGroup, excluded}`, two CURATION_FIELDS (4 → 6) and two whitelisted fields (14 → 16), both nullable/defaulted (`versionGroup: null`, `excluded: false`) so every shipped `data.json` validates unchanged — no data migration.

**The gate gains one rule, though, so this is additive *data* but not additive *logic*.** A retained-excluded Entry stays in `data.json` and therefore hits `parseSeries`, which today requires a non-empty `recommendedReason` and an integer `recommendedOrder` on every Entry. A *newly*-excluded Entry (a redundant version, a research false positive) never passes through the Order phase, so it has neither — and without a rule the gate would reject the whole publish on exactly the cases this feature exists for. The gate therefore **exempts excluded Entries from the `recommendedReason`/`recommendedOrder` requirement** and sorts null-order Entries last (the same nulls-last treatment `chronologicalOrder` already gets; excluded Entries are Shell-filtered anyway, so their relative order is irrelevant). This is the semantically correct rule — an excluded Entry has no place in the recommended order, so demanding a reason for its placement is nonsensical — and it is preferred over synthesizing placeholder reason/order values, which would keep the gate untouched at the cost of writing dishonest curation data into `data.json`. Research only **flags** group membership and redundancy (in the alignment review the maintainer already confirms) and never writes `excluded`; the maintainer decides inclusion on a version card whose N-member interaction is UI scratch, deliberately outside this schema decision.

**Relationship to ADR-0007.** This amends, not supersedes. ADR-0007's core holds unchanged: every version is its own distinct Entry, surfaced for human review, never silently merged or dropped. What changes is the two mechanisms 0007 left implicit — "the relationship is resolved by curation with no schema link" becomes a minimal durable group key, and "omission is by absence" becomes retained-exclusion — because absence is lossy and that lossiness *is* the bug.

## Considered options

- **A sidecar / tombstone list for exclusions (keep `entries` "clean")** — rejected. It keeps the read path untouched, but the omitted work's title and sources are then *not* in `entries`, so `update-series` alignment has no anchor to match a re-discovered candidate against — defeating the whole point. It would also duplicate entry-shaped data in a parallel structure (drift risk) and force the wizard to reconcile two lists on every update. Retained-exclusion keeps one list, one schema, and one alignment path; the Shell-side cost is a single filter at the one chokepoint.
- **A pairwise `versionOf` / `supersedes` pointer instead of a group key** — rejected. RE1's 1996 / 2002 / 2015 triple breaks it: the 2015 remaster is a remaster *of* the 2002 remake, so a pointer must either encode a chain the wizard has to topologically sort, or collapse to two members both pointing at the 1996 original (losing the nuance). A shared group key holds N members as a set with no traversal.
- **A durable `versionRole` enum (original / remake / remaster / port)** — rejected as speculative schema (the exact warning ADR-0007 raised). Redundancy detection needs only group membership; intra-group ordering comes from `releaseDate`; card labels are fine as research scratch. Persisting the role buys nothing the durable group key plus dates don't already give, and it is cheap to add later if a reader-facing need appears.
- **Status quo: scratch `_pairings` by title-matching + omission-by-absence** — rejected. This is what produced the four gaps: omissions leave no trace and re-surface forever; title-matching is defeated by decorated titles and a publish-stripped `versionNote`; >2-version stacks are unsupported; and research cannot flag redundancy against versions it has no durable handle on.
